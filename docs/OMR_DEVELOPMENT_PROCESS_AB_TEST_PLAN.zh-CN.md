# OMR 开发过程能力 A/B 测试方案

> 目标：验证 OMR 是否改善软件开发过程的可控性、完整性、可恢复性和交付质量。
> 执行者：Reasonix Agent。
> 约束：只做测试和报告，不修改 OMR 生产代码。

## 1. 测试目标

本实验比较 OMR 与 Native 在完整开发流程中的差异：

- 需求理解和拆解是否更完整；
- 是否减少遗漏步骤、返工和无关文件修改；
- 是否更稳定地实现、测试和修复；
- 是否更好地处理失败、中断和恢复；
- 是否提供更完整、可复核的证据；
- 是否降低人工纠偏，同时保持合理成本和耗时。

“最终产品功能更多”只是一个结果指标，不是全部结论。

## 2. 一句话产品任务

每次运行只发送一句产品需求，不提供技术方案或实施步骤：

~~~text
请在当前空项目中构建一个可运行的个人任务看板：用户可以创建、编辑、删除和完成任务，任务支持标题、描述、优先级、截止日期和标签，提供搜索与筛选，数据需要持久化，项目必须包含测试、启动说明和使用文档；完成后运行测试并汇报交付证据。
~~~

正式实验至少使用 3 个同等难度任务：

- 个人任务看板；
- 团队费用台账；
- 个人知识卡片库。

三条需求必须在首轮运行前冻结。

## 3. A/B 环境

### A 组：OMR

- 使用固定版本 OMR；
- 加载 OMR system prompt 和项目 Profile；
- 保存 manifest、配置差异、Profile 和 doctor 证据。

### B 组：Native

- 使用同一 Reasonix、模型、采样参数和项目模板；
- 不安装 OMR，不加载 OMR Prompt/Profile；
- 保存 Native 配置和工具快照。

两组使用独立目录、独立 Session、相同 Git 初始提交和相同权限。

## 4. 开发阶段记录

每次运行都记录以下阶段的开始/结束时间、输出和证据：

| 阶段 | 观察点 |
|---|---|
| P0 需求理解 | 实体、流程、约束、验收条件是否正确 |
| P1 计划拆解 | 任务、依赖、完成定义是否完整 |
| P2 方案设计 | 数据模型、接口、错误处理和取舍 |
| P3 实现 | 是否按计划推进、是否修改无关文件 |
| P4 测试 | 是否主动编写测试并覆盖边界 |
| P5 失败修复 | 是否定位根因、最小修复、回归验证 |
| P6 Review | 是否检查安全、范围和证据 |
| P7 交付 | README、启动命令、测试结果和限制 |

## 5. 重复与中断

- 每个产品每组至少运行 3 次，预算允许时运行 5 次；
- A/B 先后顺序由预先登记的随机种子决定；
- 每次使用全新的项目副本和 Session；
- 每组至少 1 次在 P3 或 P4 阶段人为中断并恢复；
- 中断点和重试规则必须预先登记；
- 不因某组失败而给另一组追加提示。

## 6. 评分与客观指标

每次运行满分 100 分，并隐藏 OMR/Native 标签进行盲评：

| 维度 | 分值 |
|---|---:|
| 需求覆盖 | 15 |
| 计划完整性 | 15 |
| 执行纪律 | 15 |
| 测试与验证 | 15 |
| 失败处理 | 10 |
| 交付质量 | 15 |
| 证据可审计性 | 10 |
| 安全与约束 | 5 |

额外记录：需求遗漏数、验收条件遗漏数、返工次数、工具调用次数、重试次数、人工纠偏次数、无关 diff、首次/最终测试通过率、恢复成功率、耗时、Token 和成本。

自动验收至少检查：

- 构建、核心功能和测试命令成功；
- 重启后数据仍存在；
- 非法输入不会导致崩溃；
- diff 只在任务范围内；
- 无密钥泄露和危险命令；
- README 能指导陌生用户启动和验证。

## 7. 故障恢复子实验

单独测试过程韧性：

1. 在实现完成但测试未完成时中断；
2. 在测试失败后中断；
3. 使用同一 Session 恢复；
4. 检查是否识别已完成步骤；
5. 检查重复修改、遗漏测试和额外人工介入。

恢复结果不得与普通运行混合统计。

## 8. 原始证据

每次运行保存：

~~~text
artifacts/development-process-ab/<run-id>/
  metadata.json
  prompt.txt
  plan.txt
  stdout.txt
  stderr.txt
  events.jsonl
  phase-timeline.json
  before-tree.txt
  after-tree.txt
  diff.patch
  test-output.txt
  acceptance.json
  blind-score.json
  delivery-report.md
~~~

metadata.json 必须记录 Reasonix、OMR、模型、项目模板、Git commit、采样参数、执行顺序、退出码、耗时、Token、成本和重试次数。不得提交 API Key 或完整敏感模型输出。

## 9. 统计和成功标准

对每个产品和全部产品分别报告：

- OMR/Native 平均过程分数、中位数和标准差；
- 配对分数差值及 95% 置信区间；
- OMR 胜/平/负比例；
- 需求遗漏、返工、无关修改、人工纠偏和恢复差异；
- 测试、证据、成本和耗时差异；
- 去掉任意一个产品后的敏感性分析。

只有同时满足以下条件，才能声称“OMR 改善了开发过程”：

1. 平均过程分数比 Native 高至少 8/100；
2. 配对差值 95% 置信区间下界大于 0；
3. 至少两个产品任务方向一致；
4. OMR 胜率至少 60%；
5. 安全、误修改和核心功能指标不劣于 Native；
6. 中断恢复或证据完整性至少有一项出现可重复改善。

只改善成本/速度时称为效率改善；只改善安装/配置时称为工程治理价值，不能统称为模型质量提升。未达标准时必须写“未证明 OMR 改善开发过程”。

## 10. Reasonix Agent 交付物

生成：

~~~text
docs/OMR_DEVELOPMENT_PROCESS_AB_TEST_REPORT.zh-CN.md
~~~

报告必须包含环境冻结、任务原文、随机化顺序、阶段时间线、全部运行记录、盲评分、自动验收、恢复实验、失败归因、成本对比和限制条件。

完成后运行：

~~~bash
git diff --check
go test ./...
go vet ./...
~~~

本测试只生成 Fixture、验收脚本、证据和报告，不修改 OMR 生产代码。
