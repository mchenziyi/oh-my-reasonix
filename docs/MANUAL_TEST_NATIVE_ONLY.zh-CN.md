# 不安装 OMR 的 Native Reasonix 人工测试清单

本清单用于建立原生 Reasonix 基线。执行期间不要安装、复制或引用 oh-my-reasonix 的 Prompt、Profile、配置和事件转发命令。

## 1. 准备干净项目

从与 OMR 组相同的初始快照复制一份项目：

~~~bash
cp -R /path/to/source-project /tmp/omr-ab-native
cd /tmp/omr-ab-native
reasonix --version
~~~

确认：

~~~bash
test ! -e .reasonix/omr/manifest.lock.yaml
git status --short
~~~

如果项目已有 OMR 文件，换用干净快照，不要手工删除后继续测试。

## 2. 原生能力记录

只使用 Reasonix 原生命令和内置 Profile：

~~~bash
reasonix subagent list
reasonix --help
~~~

记录 Reasonix 版本、内置 Profile 列表、是否支持 Session 恢复、Hook 查询、Task 查询和结构化事件流。某项不支持就记录“不支持”，不要从人类可读输出猜测机器状态。

## 3. 与 OMR 组相同的任务

逐字发送以下提示词，并保存完整会话输出：

### N1：项目探索

~~~text
请只读分析当前项目：列出主要模块、程序入口、测试入口、关键依赖和潜在高风险区域。不要修改文件。结论必须引用实际文件路径，并区分已确认事实与推测。
~~~

### N2：需求拆解

~~~text
请把“为当前项目增加一个可配置的健康检查命令”拆成最小实施任务。先列假设、影响文件、风险、验收条件和测试命令，再开始修改。暂时不要修改代码。
~~~

### N3：故障定位

~~~text
请定位当前项目中一个最值得优先修复的测试或构建问题。先收集证据并给出根因，再提出最小修复方案；不要为了证明结论而大范围重构。
~~~

### N4：Review

~~~text
请对当前项目做一次只读代码 Review，重点检查输入校验、路径越界、敏感信息泄露、错误处理和测试缺口。每个问题给出严重级别、文件路径、证据和修复建议；没有证据不要猜测。
~~~

### N5：恢复

让 N1 或 N2 在中途停止，再发送：

~~~text
请从上次中断的位置继续。先说明已完成和未完成的步骤，不要重复已完成的工作；继续前先检查工作区状态。
~~~

### N6：原生运行记录（可选）

使用当前 Reasonix 版本支持的原生命令完成与 OMR 组 T6 相同的只读任务。若没有 JSONL 事件流，保存终端输出并记录“不支持”。
