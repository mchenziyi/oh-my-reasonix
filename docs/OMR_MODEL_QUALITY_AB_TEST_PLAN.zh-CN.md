# OMR 模型执行质量 A/B 测试方案

> 版本：v1.0  
> 用途：交给 Reasonix Agent 执行，验证 OMR 是否提升模型的实际任务执行质量。  
> 约束：本方案只做实验、采集证据和生成报告，不修改 OMR 或被测项目代码。

## 1. 实验目标

验证在相同 Reasonix 版本、相同模型、相同项目快照和相同任务提示下：

- OMR 组是否比 Native 组得到更高的任务质量分数；
- OMR 是否降低失败率、返工次数和人工纠偏次数；
- OMR 的收益是否来自可重复的工程流程，而不是单次模型随机波动；
- OMR 的收益是否伴随不可接受的成本、耗时或误修改增加。

不能用本实验证明的内容：

- OMR 对所有模型、所有项目或所有 Reasonix 版本都有效；
- OMR 提升了模型底层推理能力；
- OMR 的 Prompt/Profile 设计优于所有其他编排方案。

## 2. 实验假设与成功标准

### 2.1 假设

- H0：OMR 与 Native 的执行质量没有统计显著差异。
- H1：OMR 的平均执行质量高于 Native。

### 2.2 主要成功标准

只有同时满足以下条件，才能写“本实验支持 OMR 提升模型执行质量”：

1. OMR 与 Native 的配对平均质量差值 `OMR - Native >= 8/100`；
2. 配对差值的 95% 置信区间下界大于 0；
3. OMR 的任务胜率（OMR 分数更高的配对比例）不低于 60%；
4. OMR 没有触发安全回归：误修改率、越权写入率均不得高于 Native；
5. 主要结论在去除单个任务后仍保持方向一致（leave-one-task-out）；
6. 两组任务完成率均至少 90%，否则只能报告“实验不可判定”。

若只满足质量差值但不满足统计和安全条件，结论必须写为“存在趋势，证据不足”。

## 3. 固定实验环境

执行前记录并冻结以下信息：

| 项目 | 要求 |
|---|---|
| Reasonix 版本 | 使用同一版本，建议当前稳定版本；记录 `reasonix --version` |
| OMR 版本 | 记录 Git commit、Release 或构建哈希 |
| 模型 | 两组必须完全相同，记录模型名和参数 |
| 采样参数 | temperature、top_p、seed（若支持）必须一致 |
| 项目 | 使用只读副本或完全相同的 Git commit |
| 工具 | 两组都记录可用工具、Profile 和 MCP 配置 |
| 网络 | 两组使用相同网络策略；失败需标记基础设施阻断 |
| 时间 | 尽量在同一时间窗口执行，记录开始和结束时间 |

禁止在实验中途：

- 升级 Reasonix、模型、OMR 或 MCP；
- 修改任务提示、评分标准或项目快照；
- 将 A 组输出、事件或修复结果泄漏给 B 组；
- 因 A 组失败而临时修改任务后重跑；
- 删除失败记录或只报告成功样本。

## 4. A/B 组定义

### A 组：OMR

1. 在独立项目副本执行 OMR 安装/升级。
2. 运行 `omr doctor --json`、`omr profile list --json`。
3. 保存 system prompt、manifest、配置差异和 Profile 清单。
4. 使用 OMR 生成的项目配置执行任务。

### B 组：Native

1. 从同一项目快照创建独立副本。
2. 不安装 OMR，不加载 OMR system prompt 或 OMR Profile。
3. 使用同一 Reasonix、模型、工具和任务提示执行任务。
4. 保存 Native 配置和工具可用性快照。

两组都必须使用独立工作目录、独立 Session 和独立输出目录。

## 5. 任务集设计

至少选择 12 个任务，覆盖下列类别；每个任务必须有明确的输入快照、成功条件和自动验证方式。

| 编号 | 类别 | 目标 |
|---|---|---|
| Q1 | 项目探索 | 找出入口、依赖、运行方式并给出文件证据 |
| Q2 | 需求拆解 | 将自然语言需求拆成可执行任务和验收条件 |
| Q3 | Bug 定位 | 根据失败测试定位根因，不修改无关文件 |
| Q4 | 小型实现 | 完成一个边界清晰的小功能并补测试 |
| Q5 | 回归修复 | 修复一个已有失败测试，保持其他测试通过 |
| Q6 | 重构建议 | 提出低风险重构方案并说明不变量 |
| Q7 | 安全 Review | 识别高风险点并给出文件/行号证据 |
| Q8 | 测试设计 | 为未覆盖模块补充测试计划或测试代码 |
| Q9 | 文档同步 | 根据代码变化更新准确文档，不臆造接口 |
| Q10 | 多步骤任务 | 需求→实现→测试→总结的完整闭环 |
| Q11 | 中断恢复 | 在人为中断后从正确步骤继续，不重复已完成工作 |
| Q12 | 只读约束 | 明确要求只读，验证不产生任何工作区修改 |

其中 Q4、Q5、Q8、Q10 应使用可自动验证的实现任务；Q1、Q2、Q3、Q6、Q7、Q9、Q11、Q12 使用证据和工作区快照验证。

任务提示必须在实验开始前冻结，放入：

```text
docs/ab-fixtures/model-quality/tasks.yaml
```

每个任务至少包含：`id`、`prompt`、`project_commit`、`read_only`、`expected_checks`、`scoring_notes`。

## 6. 重复与随机化

- 每个任务每组至少运行 5 次；12 个任务共 120 次运行（12 × 5 × 2）。
- 每个配对使用相同项目快照和相同任务提示。
- A/B 执行顺序使用预先生成的随机种子随机化，不能固定先 A 后 B。
- 若模型支持 seed，使用 5 个预先登记的 seed；不支持时记录实际请求参数。
- 每个配对运行之间清理 Session、缓存和临时输出，但不能改变项目快照。
- 不允许根据某次输出质量决定是否增加重试；重试规则必须预先写入任务定义。

## 7. 质量评分（盲评）

评分者在隐藏组别和运行编号后评分，不能看到 `OMR`、`Native` 等标签。

每次运行 100 分：

| 维度 | 分值 | 评分要点 |
|---|---:|---|
| 正确性 | 35 | 结论/代码是否正确，自动测试是否通过 |
| 完整性 | 20 | 是否覆盖任务要求和边界条件 |
| 证据质量 | 15 | 文件路径、行号、命令输出是否可复核 |
| 约束遵守 | 15 | 是否遵守只读、范围、格式和安全要求 |
| 可执行性 | 10 | 建议或代码是否能直接使用 |
| 表达清晰度 | 5 | 结构清晰、事实与推测分离 |

自动指标优先于主观评分：编译、测试、静态检查、工作区 diff、越权写入和命令退出码必须由脚本记录。

每个任务由两名独立评分者评分；分差超过 10 分时进行第三人复核。评分者一致性使用 Cohen's kappa 或简单一致率报告。

## 8. 必须采集的原始证据

每次运行保存以下文件，禁止只保存最终摘要：

```text
artifacts/model-quality-ab/<run-id>/
  metadata.json
  prompt.txt
  stdout.txt
  stderr.txt
  events.jsonl          # 若宿主支持
  before-tree.txt
  after-tree.txt
  diff.patch
  checks.json
  score.json
```

`metadata.json` 至少包含：组别、任务 ID、重复编号、随机种子、Reasonix/OMR/模型版本、项目 commit、开始结束时间、退出码、重试次数、token、成本和耗时。

任何 API Key、完整敏感模型输出、用户隐私和未授权项目内容不得写入报告或 Git。

## 9. 分类规则

每次运行只能有一个最终分类：

- `pass`：任务完成且自动检查、证据和约束均通过；
- `task_failure`：模型未完成任务或自动检查失败；
- `omr_defect`：仅当能用最小复现证明 OMR 行为导致失败；
- `host_defect`：Reasonix 宿主接口或运行时行为导致失败；
- `blocked_infrastructure`：权限、网络、服务不可用等外部阻断；
- `evidence_incomplete`：任务可能完成，但缺少足够原始证据，不能用于质量结论。

不能因为 A 组失败就自动归因 OMR，也不能因为 B 组成功就自动归因 Native 优势。

## 10. 统计分析

以每个任务的 5 个配对差值为基本单位，报告：

1. 每任务 OMR/Native 均值、中位数、标准差；
2. 配对平均差值及 95% bootstrap 置信区间；
3. OMR 胜/平/负比例；
4. 完成率、自动测试通过率、误修改率、人工纠偏次数；
5. token、成本、耗时和重试次数；
6. 按任务类别分层结果；
7. leave-one-task-out 敏感性分析；
8. 失败样本和基础设施阻断单独统计，不得从分母删除后隐瞒。

推荐使用配对 bootstrap；样本不足时同时报告原始数据，不声称统计显著。

## 11. 报告结构与验收

最终报告写入：

```text
docs/OMR_MODEL_QUALITY_AB_TEST_REPORT.zh-CN.md
```

必须包含：

1. 执行摘要；
2. 环境和版本冻结记录；
3. 任务清单和随机化顺序；
4. A/B 原始运行统计；
5. 盲评方法和评分一致性；
6. 自动验证结果；
7. 统计结果和置信区间；
8. 失败、阻断和缺失证据清单；
9. OMR 与宿主问题的归因；
10. 成本/耗时/安全回归；
11. 限制条件；
12. 最终结论和是否支持 H1。

报告不得使用“明显更好”“全面提升”等无法由数据支持的表述。

## 12. 执行门禁

Reasonix Agent 完成后必须执行：

```bash
git diff --check
go test ./...
go vet ./...
```

如果只生成实验 Fixture 和报告，不应修改生产代码。除非用户另行授权，不要提交 API Key、完整模型输出、临时目录或大体积事件日志。

交付前确认：

- [ ] 任务提示和评分标准在首个运行前已冻结；
- [ ] A/B 使用相同项目 commit、模型和工具条件；
- [ ] 每个任务有 5 个配对重复；
- [ ] 所有失败和阻断都保留原始证据；
- [ ] 统计分母和缺失值处理透明；
- [ ] 结论区分模型质量、工程治理和宿主能力；
- [ ] 未达到成功标准时明确写“未证明提升”。
